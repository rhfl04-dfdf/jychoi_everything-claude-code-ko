# 개발 워크플로우

> 이 파일은 [common/git-workflow.md](./git-workflow.md)를 확장하여, git 작업 이전에 수행되는 전체 기능 개발 프로세스를 다룹니다.

기능 구현 워크플로우는 개발 파이프라인을 설명합니다: 리서치, 계획, TDD, 코드 리뷰, 그리고 git 커밋.

## 기능 구현 워크플로우

0. **리서치 및 재사용** _(새로운 구현 전 필수)_
   - **GitHub 코드 검색 우선:** 새로 작성하기 전에 `gh search repos`와 `gh search code`를 실행하여 기존 구현, 템플릿, 패턴을 찾습니다.
   - **라이브러리 문서 두 번째:** 구현하기 전에 Context7 또는 주요 벤더 문서를 사용하여 API 동작, 패키지 사용법, 버전별 세부 사항을 확인합니다.
   - **Exa는 위 두 가지가 충분하지 않을 때만:** GitHub 검색과 공식 문서 이후에도 부족할 때 Exa를 사용하여 더 넓은 웹 리서치 또는 발견을 수행합니다.
   - **패키지 레지스트리 확인:** 유틸리티 코드를 작성하기 전에 npm, PyPI, crates.io 및 기타 레지스트리를 검색합니다. 직접 구현보다 검증된 라이브러리를 선호합니다.
   - **적용 가능한 구현 탐색:** 문제의 80% 이상을 해결하고 포크, 포팅 또는 래핑할 수 있는 오픈소스 프로젝트를 찾습니다.
   - 요구사항을 충족하는 경우 새 코드 작성보다 검증된 접근 방식을 채택하거나 포팅하는 것을 선호합니다.

1. **먼저 계획 수립**
   - **planner** 에이전트를 사용하여 구현 계획 작성
   - 코딩 전에 계획 문서 생성: PRD, architecture, system_design, tech_doc, task_list
   - 의존성 및 위험 요소 파악
   - 단계별로 분해

2. **TDD 접근 방식**
   - **tdd-guide** 에이전트 사용
   - 테스트 먼저 작성 (RED)
   - 테스트를 통과하도록 구현 (GREEN)
   - 리팩터링 (IMPROVE)
   - 80% 이상 커버리지 확인

3. **코드 리뷰**
   - 코드 작성 직후 **code-reviewer** 에이전트 사용
   - CRITICAL 및 HIGH 이슈 처리
   - 가능한 경우 MEDIUM 이슈 수정

4. **커밋 & 푸시**
   - 상세한 커밋 메시지 작성
   - Conventional Commits 형식 준수
   - 커밋 메시지 형식 및 PR 프로세스는 [git-workflow.md](./git-workflow.md) 참조
