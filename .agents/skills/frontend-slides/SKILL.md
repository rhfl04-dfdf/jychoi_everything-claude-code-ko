# 프론트엔드 슬라이드

브라우저에서 완전히 실행되는 의존성 없는 애니메이션 풍부한 HTML 프레젠테이션을 만듭니다.

zarazhangrui의 작업에서 소개된 시각적 탐색 접근법에서 영감을 받았습니다 (크레딧: @zarazhangrui).

## 활성화 시점

- 발표용, 피치덱, 워크숍 덱, 내부 프레젠테이션 생성
- `.ppt` 또는 `.pptx` 슬라이드를 HTML 프레젠테이션으로 변환
- 기존 HTML 프레젠테이션의 레이아웃, 모션, 타이포그래피 개선
- 디자인 선호도를 모르는 사용자와 함께 프레젠테이션 스타일 탐색

## 필수 요건

1. **의존성 없음**: 기본적으로 인라인 CSS와 JS가 포함된 단일 자기완결 HTML 파일.
2. **뷰포트 핏은 필수**: 모든 슬라이드는 내부 스크롤 없이 하나의 뷰포트 안에 들어와야 함.
3. **보여주기, 설명하지 않기**: 추상적인 스타일 설문 대신 시각적 미리보기 사용.
4. **독특한 디자인**: 일반적인 보라색 그라디언트, 흰 배경의 Inter 폰트, 템플릿 같은 덱 지양.
5. **프로덕션 품질**: 코드는 주석 처리, 접근성, 반응형, 성능을 갖춰야 함.

생성 전 뷰포트 안전 CSS 베이스, 밀도 제한, 프리셋 카탈로그, CSS 주의사항에 대해 `STYLE_PRESETS.md`를 읽으세요.

## 워크플로우

### 1. 모드 감지

하나의 경로를 선택:
- **새 프레젠테이션**: 사용자가 주제, 메모, 또는 전체 초안 보유
- **PPT 변환**: 사용자가 `.ppt` 또는 `.pptx` 보유
- **개선**: 사용자가 이미 HTML 슬라이드를 보유하고 개선을 원함

### 2. 콘텐츠 파악

최소한으로 필요한 것만 질문:
- 목적: 피치, 교육, 컨퍼런스 발표, 내부 업데이트
- 길이: 짧음 (5-10), 중간 (10-20), 길음 (20+)
- 콘텐츠 상태: 완성된 내용, 대략적인 메모, 주제만

사용자가 콘텐츠를 가지고 있다면, 스타일 작업 전에 붙여넣기를 요청하세요.

### 3. 스타일 발견

기본적으로 시각적 탐색을 사용합니다.

사용자가 이미 원하는 프리셋을 알고 있다면, 미리보기를 건너뛰고 직접 사용하세요.

그렇지 않으면:
1. 덱이 주어야 할 느낌을 물어보세요: 인상적, 활기차게, 집중적, 영감적.
2. `.ecc-design/slide-previews/`에 **3개의 단일 슬라이드 미리보기 파일** 생성.
3. 각 미리보기는 자기완결적이어야 하고, 타이포그래피/색상/모션을 명확하게 보여줘야 하며, 슬라이드 콘텐츠는 대략 100줄 이내여야 함.
4. 사용자에게 유지할 미리보기나 혼합할 요소를 물어보세요.

스타일을 무드에 매핑할 때 `STYLE_PRESETS.md`의 프리셋 가이드를 사용하세요.

### 4. 프레젠테이션 작성

출력:
- `presentation.html`
- `[presentation-name].html`

덱에 추출되거나 사용자 제공 이미지가 포함된 경우에만 `assets/` 폴더 사용.

필수 구조:
- 시맨틱 슬라이드 섹션
- `STYLE_PRESETS.md`의 뷰포트 안전 CSS 베이스
- 테마 값을 위한 CSS 커스텀 프로퍼티
- 키보드, 휠, 터치 네비게이션을 위한 프레젠테이션 컨트롤러 클래스
- 등장 애니메이션을 위한 Intersection Observer
- 모션 감소 지원

### 5. 뷰포트 핏 적용

이것을 하드 게이트로 처리하세요.

규칙:
- 모든 `.slide`는 `height: 100vh; height: 100dvh; overflow: hidden;` 사용
- 모든 타입과 간격은 `clamp()`로 스케일
- 콘텐츠가 맞지 않으면 여러 슬라이드로 분리
- 읽기 어려운 크기로 텍스트를 줄여서 오버플로우 해결 금지
- 슬라이드 내부에 스크롤바 허용 금지

`STYLE_PRESETS.md`의 밀도 제한과 필수 CSS 블록을 사용하세요.

### 6. 검증

다음 크기에서 완성된 덱 확인:
- 1920x1080
- 1280x720
- 768x1024
- 375x667
- 667x375

브라우저 자동화가 가능하다면 슬라이드 오버플로우가 없고 키보드 네비게이션이 동작하는지 확인하세요.

### 7. 납품

인계 시:
- 사용자가 유지하길 원하지 않는다면 임시 미리보기 파일 삭제
- 유용한 경우 플랫폼에 맞는 오프너로 덱 열기
- 파일 경로, 사용된 프리셋, 슬라이드 수, 쉬운 테마 커스터마이징 포인트 요약

현재 OS에 맞는 오프너 사용:
- macOS: `open file.html`
- Linux: `xdg-open file.html`
- Windows: `start "" file.html`

## PPT / PPTX 변환

PowerPoint 변환 시:
1. `python3`와 `python-pptx`를 사용하여 텍스트, 이미지, 노트를 추출하는 것을 선호.
2. `python-pptx`가 없다면 설치 여부를 묻거나 수동/내보내기 기반 워크플로우로 대체.
3. 슬라이드 순서, 발표자 노트, 추출된 에셋 보존.
4. 추출 후 새 프레젠테이션과 동일한 스타일 선택 워크플로우 실행.

변환을 크로스 플랫폼으로 유지하세요. Python으로 할 수 있는 작업에는 macOS 전용 도구에 의존하지 마세요.

## 구현 요건

### HTML / CSS

- 사용자가 멀티 파일 프로젝트를 명시적으로 원하지 않는 한 인라인 CSS와 JS 사용.
- 폰트는 Google Fonts 또는 Fontshare에서.
- 분위기 있는 배경, 강한 타입 계층, 명확한 시각적 방향을 선호.
- 일러스트레이션 대신 추상적인 도형, 그라디언트, 격자, 노이즈, 기하학 사용.

### JavaScript

포함 사항:
- 키보드 네비게이션
- 터치 / 스와이프 네비게이션
- 마우스 휠 네비게이션
- 진행 표시기 또는 슬라이드 인덱스
- 등장 애니메이션 트리거

### 접근성

- 시맨틱 구조 사용 (`main`, `section`, `nav`)
- 읽기 쉬운 대비 유지
- 키보드 전용 네비게이션 지원
- `prefers-reduced-motion` 존중

## 콘텐츠 밀도 제한

사용자가 명시적으로 더 조밀한 슬라이드를 요청하고 가독성이 유지되는 경우를 제외하고 다음 최대값 사용:

| 슬라이드 유형 | 제한 |
|------------|-------|
| 타이틀 | 헤딩 1개 + 서브타이틀 1개 + 선택적 태그라인 |
| 콘텐츠 | 헤딩 1개 + 불릿 4-6개 또는 짧은 단락 2개 |
| 기능 그리드 | 카드 최대 6개 |
| 코드 | 최대 8-10줄 |
| 인용 | 인용 1개 + 출처 |
| 이미지 | 뷰포트에 맞게 제한된 이미지 1개 |

## 안티패턴

- 시각적 정체성 없는 일반적인 스타트업 그라디언트
- 의도적인 에디토리얼이 아닌 경우 시스템 폰트 덱
- 긴 불릿 목록
- 스크롤이 필요한 코드 블록
- 짧은 화면에서 깨지는 고정 높이 콘텐츠 박스
- `-clamp(...)`와 같은 유효하지 않은 부정된 CSS 함수

## 납품 체크리스트

- 프레젠테이션이 브라우저의 로컬 파일에서 실행됨
- 모든 슬라이드가 스크롤 없이 뷰포트에 맞음
- 스타일이 독특하고 의도적
- 애니메이션이 의미 있고 과도하지 않음
- 모션 감소가 존중됨
- 파일 경로와 커스터마이징 포인트가 인계 시 설명됨
