# Ralphinho RFC 파이프라인

[humanplane](https://github.com/humanplane) 스타일의 RFC 분해 패턴과 다중 단위 오케스트레이션 워크플로우에서 영감을 받았습니다.

기능이 단일 에이전트 패스로는 처리하기에 너무 크고, 독립적으로 검증 가능한 작업 단위로 분할해야 할 때 이 skill을 사용하세요.

## 파이프라인 단계

1. RFC 접수
2. DAG 분해
3. 단위 할당
4. 단위 구현
5. 단위 검증
6. 병합 큐 및 통합
7. 최종 시스템 검증

## 단위 명세 템플릿

각 작업 단위에는 다음이 포함되어야 합니다:
- `id`
- `depends_on`
- `scope`
- `acceptance_tests`
- `risk_level`
- `rollback_plan`

## 복잡도 등급

- Tier 1: 독립적인 파일 편집, 결정론적 테스트
- Tier 2: 다중 파일 동작 변경, 중간 수준의 통합 위험
- Tier 3: 스키마/인증/성능/보안 변경

## 단위별 품질 파이프라인

1. 조사
2. 구현 계획
3. 구현
4. 테스트
5. 리뷰
6. 병합 준비 보고서

## 병합 큐 규칙

- 미해결 의존성 실패가 있는 단위는 절대 병합하지 않는다.
- 항상 단위 브랜치를 최신 통합 브랜치에 리베이스한다.
- 큐에 올라간 병합 후에는 통합 테스트를 다시 실행한다.

## 복구

단위가 중단될 경우:
- 활성 큐에서 제거
- 발견 사항 스냅샷 저장
- 범위를 좁혀 단위 재생성
- 업데이트된 제약 조건으로 재시도

## 출력물

- RFC 실행 로그
- 단위 성적표
- 의존성 그래프 스냅샷
- 통합 위험 요약
