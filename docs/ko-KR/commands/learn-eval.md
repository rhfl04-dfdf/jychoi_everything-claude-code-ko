# /learn-eval - 추출, 평가, 그 다음 저장

`/learn`을 품질 게이트, 저장 위치 결정, skill 파일 작성 전 지식 배치 인식으로 확장합니다.

## 추출할 내용

다음을 찾습니다:

1. **에러 해결 패턴** — 근본 원인 + 수정 + 재사용 가능성
2. **디버깅 기법** — 비직관적인 단계, 도구 조합
3. **우회 방법** — 라이브러리 특이 사항, API 제한, 버전별 수정
4. **프로젝트별 패턴** — 컨벤션, 아키텍처 결정, 통합 패턴

## 프로세스

1. 추출 가능한 패턴에 대해 세션 검토
2. 가장 가치 있고 재사용 가능한 인사이트 식별

3. **저장 위치 결정:**
   - 질문: "이 패턴이 다른 프로젝트에서도 유용할까?"
   - **글로벌** (`~/.claude/skills/learned/`): 2개 이상의 프로젝트에서 사용 가능한 일반 패턴 (bash 호환성, LLM API 동작, 디버깅 기법 등)
   - **프로젝트** (현재 프로젝트의 `.claude/skills/learned/`): 프로젝트별 지식 (특정 설정 파일의 특이 사항, 프로젝트별 아키텍처 결정 등)
   - 확실하지 않을 때는 글로벌 선택 (글로벌 → 프로젝트 이동이 반대보다 쉬움)

4. 다음 형식을 사용하여 skill 파일 초안 작성:

```markdown
---
name: pattern-name
description: "130자 이하"
user-invocable: false
origin: auto-extracted
---

# [설명적인 패턴 이름]

**추출일:** [날짜]
**컨텍스트:** [적용되는 시점에 대한 간략한 설명]

## 문제
[이것이 해결하는 문제 - 구체적으로]

## 해결책
[패턴/기법/우회 방법 - 코드 예시 포함]

## 사용 시점
[트리거 조건]
```

5. **품질 게이트 — 체크리스트 + 종합 판정**

   ### 5a. 필수 체크리스트 (실제 파일을 읽어서 확인)

   초안을 평가하기 전에 다음을 **모두** 실행합니다:

   - [ ] `~/.claude/skills/`와 관련 프로젝트 `.claude/skills/` 파일을 키워드로 grep하여 내용 중복 확인
   - [ ] MEMORY.md (프로젝트와 글로벌 모두) 중복 확인
   - [ ] 기존 skill에 추가하는 것으로 충분한지 고려
   - [ ] 이것이 일회성 수정이 아닌 재사용 가능한 패턴인지 확인

   ### 5b. 종합 판정

   체크리스트 결과와 초안 품질을 종합하여 다음 중 **하나** 선택:

   | 판정 | 의미 | 다음 액션 |
   |---------|---------|-------------|
   | **저장** | 고유하고, 구체적이며, 잘 범위가 정해짐 | 6단계로 진행 |
   | **개선 후 저장** | 가치 있지만 개선 필요 | 개선 사항 나열 → 수정 → 재평가 (한 번) |
   | **[X]에 흡수** | 기존 skill에 추가해야 함 | 대상 skill과 추가 사항 표시 → 6단계 |
   | **삭제** | 사소하거나, 중복되거나, 너무 추상적 | 이유를 설명하고 중단 |

   **가이드라인 차원** (판정을 위한 참고 사항, 점수 아님):

   - **구체성 및 실행 가능성**: 즉시 사용 가능한 코드 예시나 커맨드 포함
   - **범위 적합성**: 이름, 트리거 조건, 내용이 단일 패턴에 맞게 정렬됨
   - **고유성**: 기존 skill에서 다루지 않는 가치 제공 (체크리스트 결과 기반)
   - **재사용 가능성**: 미래 세션에서 현실적인 트리거 시나리오 존재

6. **판정별 확인 흐름**

   - **개선 후 저장**: 필요한 개선 사항 + 수정된 초안 + 한 번의 재평가 후 업데이트된 체크리스트/판정 제시; 수정된 판정이 **저장**이면 사용자 확인 후 저장, 그렇지 않으면 새 판정 따르기
   - **저장**: 저장 경로 + 체크리스트 결과 + 1줄 판정 이유 + 전체 초안 제시 → 사용자 확인 후 저장
   - **[X]에 흡수**: 대상 경로 + 추가 사항 (diff 형식) + 체크리스트 결과 + 판정 이유 제시 → 사용자 확인 후 추가
   - **삭제**: 체크리스트 결과 + 이유만 표시 (확인 불필요)

7. 결정된 위치에 저장/흡수

## 5단계 출력 형식

```
### 체크리스트
- [x] skills/ grep: 중복 없음 (또는: 중복 발견 → 세부 사항)
- [x] MEMORY.md: 중복 없음 (또는: 중복 발견 → 세부 사항)
- [x] 기존 skill 추가: 새 파일 적합 (또는: [X]에 추가해야 함)
- [x] 재사용 가능성: 확인됨 (또는: 일회성 → 삭제)

### 판정: 저장 / 개선 후 저장 / [X]에 흡수 / 삭제

**이유:** (판정을 설명하는 1-2문장)
```

## 설계 의도

이 버전은 이전의 5차원 수치 채점 루브릭 (구체성, 실행 가능성, 범위 적합성, 비중복성, 커버리지를 1-5점으로 채점)을 체크리스트 기반 종합 판정 시스템으로 대체합니다. 최신 frontier 모델 (Opus 4.6+)은 강력한 맥락적 판단력을 가지고 있습니다 — 풍부한 질적 신호를 수치 점수로 강제 변환하면 뉘앙스를 잃고 오해의 소지가 있는 합계를 생성할 수 있습니다. 종합적 접근 방식은 모델이 모든 요소를 자연스럽게 고려할 수 있게 하여 더 정확한 저장/삭제 결정을 생성하며, 명시적 체크리스트는 중요한 검사가 건너뛰어지지 않도록 합니다.

## 참고 사항

- 사소한 수정 (오타, 단순 구문 에러)은 추출하지 않습니다
- 일회성 이슈 (특정 API 중단 등)는 추출하지 않습니다
- 미래 세션에서 시간을 절약할 패턴에 집중합니다
- skill은 집중적으로 유지합니다 — skill당 하나의 패턴
- 판정이 흡수인 경우 새 파일을 만들지 않고 기존 skill에 추가합니다
