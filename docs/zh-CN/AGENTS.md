# Everything Claude Code (ECC) — Agent 지침

이것은 **프로덕션 수준의 AI 코딩 플러그인**으로, 16개의 전문 agent, 65개 이상의 skill, 40개의 command, 그리고 자동화 hook 워크플로우를 소프트웨어 개발에 제공합니다.

## 핵심 원칙

1. **Agent 우선** — 도메인 작업을 전문 agent에 위임
2. **테스트 주도** — 구현 전에 테스트를 먼저 작성하고, 80% 이상 커버리지 요구
3. **보안 최우선** — 보안에 절대 타협하지 않으며 모든 입력을 검증
4. **불변성** — 항상 새 객체를 생성하고 기존 객체를 절대 수정하지 않음
5. **계획 후 실행** — 코드를 작성하기 전에 복잡한 기능을 계획

## 사용 가능한 Agent

| tdd-guide | 테스트 주도 개발 | 새 기능, 버그 수정 |

## Agent 오케스트레이션

사용자 프롬프트 없이도 능동적으로 agent를 사용하세요:

* 복잡한 기능 요청 → **planner**
* 방금 작성/수정한 코드 → **code-reviewer**
* 버그 수정 또는 새 기능 → **tdd-guide**
* 아키텍처 결정 → **architect**
* 보안에 민감한 코드 → **security-reviewer**
* 멀티채널 커뮤니케이션 분류 → **chief-of-staff**
* 자율 루프 / 루프 모니터링 → **loop-operator**
* 하네스 설정 신뢰성 및 비용 → **harness-optimizer**

독립적인 작업에는 병렬 실행을 사용하세요 — 여러 agent를 동시에 시작하세요.

## 보안 가이드라인

**모든 커밋 전에:**

* 하드코딩된 시크릿 없음 (API 키, 비밀번호, 토큰)
* 모든 사용자 입력 검증 완료
* SQL 인젝션 방지 (파라미터화된 쿼리)
* XSS 방지 (새니타이즈된 HTML)
* CSRF 보호 활성화
* 인증/인가 검증 완료
* 모든 엔드포인트에 속도 제한 적용
* 오류 메시지에 민감한 데이터 미노출

**시크릿 관리:** 시크릿을 절대 하드코딩하지 마세요. 환경 변수 또는 시크릿 매니저를 사용하세요. 시작 시 필요한 시크릿을 검증하세요. 노출된 시크릿은 즉시 로테이션하세요.

**보안 문제 발견 시:** 중지 → security-reviewer agent 사용 → CRITICAL 문제 수정 → 노출된 시크릿 로테이션 → 코드베이스에서 유사한 문제 검토.

## 코딩 스타일

**불변성 (핵심):** 항상 새 객체를 생성하고 절대 수정하지 마세요. 변경 사항이 포함된 새 복사본을 반환하세요.

**파일 구성:** 적은 수의 큰 파일보다 많은 수의 작은 파일을 선호하세요. 보통 200-400줄, 최대 800줄. 타입이 아닌 기능/도메인별로 구성하세요. 높은 응집도, 낮은 결합도.

**오류 처리:** 각 레이어에서 오류를 처리하세요. UI 코드에서는 사용자 친화적인 메시지를 제공하세요. 서버 측에서는 상세한 컨텍스트를 로깅하세요. 오류를 절대 조용히 무시하지 마세요.

**입력 검증:** 시스템 경계에서 모든 사용자 입력을 검증하세요. 스키마 기반 검증을 사용하세요. 명확한 메시지와 함께 빠르게 실패하세요. 외부 데이터를 절대 신뢰하지 마세요.

**코드 품질 체크리스트:**

* 함수는 작게 (<50줄), 파일은 집중적으로 (<800줄)
* 깊은 중첩 없음 (>4단계)
* 적절한 오류 처리, 하드코딩된 값 없음
* 읽기 쉽고 잘 명명된 식별자

## 테스트 요구사항

**최소 커버리지: 80%**

테스트 유형 (모두 필수):

1. **단위 테스트** — 개별 함수, 도구, 컴포넌트
2. **통합 테스트** — API 엔드포인트, 데이터베이스 작업
3. **E2E 테스트** — 핵심 사용자 플로우

**TDD 워크플로우 (필수):**

1. 테스트를 먼저 작성 (RED) — 테스트가 실패해야 함
2. 최소 구현 작성 (GREEN) — 테스트가 통과해야 함
3. refactor (IMPROVE) — 커버리지 80% 이상 확인

트러블슈팅: 테스트 격리 확인 → mock 검증 → 구현을 수정 (테스트가 잘못된 경우가 아니라면 테스트가 아닌 구현을 수정).

## 개발 워크플로우

1. **계획** — planner agent 사용, 의존성과 리스크 식별, 단계별 진행
2. **테스트 주도 개발** — tdd-guide agent 사용, 테스트 먼저 작성, 구현 및 refactor
3. **리뷰** — 즉시 code-reviewer agent 사용, CRITICAL/HIGH 수준 이슈 해결
4. **적절한 위치에 지식 기록**
   * 개인 디버깅 노트, 선호사항 및 임시 컨텍스트 → 자동 메모리
   * 팀/프로젝트 지식 (아키텍처 결정, API 변경, 운영 매뉴얼) → 프로젝트 기존 문서 구조
   * 현재 작업에서 관련 문서 또는 코드 주석이 이미 생성된 경우, 다른 곳에 같은 정보를 중복하지 마세요
   * 명확한 프로젝트 문서 위치가 없는 경우, 새 최상위 파일을 만들기 전에 먼저 질문하세요
5. **커밋** — Conventional Commit 형식 사용, 포괄적인 PR 요약 제공

## Git 워크플로우

**커밋 형식:** `<type>: <description>` — 타입: feat, fix, refactor, docs, test, chore, perf, ci

**PR 워크플로우:** 전체 커밋 이력 분석 → 포괄적인 요약 작성 → 테스트 계획 포함 → `-u` 플래그로 push.

## 아키텍처 패턴

**API 응답 형식:** 성공 표시자, 데이터 페이로드, 오류 메시지, 페이지네이션 메타데이터가 포함된 일관된 엔벨로프.

**Repository 패턴:** 데이터 접근을 표준 인터페이스 (findAll, findById, create, update, delete) 뒤에 캡슐화. 비즈니스 로직은 저장 메커니즘이 아닌 추상 인터페이스에 의존.

**스켈레톤 프로젝트:** 실전 검증된 템플릿 검색, 병렬 agent (보안, 확장성, 관련성)로 평가, best practice 매칭 클론, 검증된 구조 내에서 반복.

## 성능

**컨텍스트 관리:** 대규모 refactor 및 다중 파일 기능의 경우, 컨텍스트 윈도우의 마지막 20%를 사용하지 마세요. 민감도가 낮은 작업 (단일 편집, 문서, 간단한 수정)은 높은 활용률을 허용할 수 있습니다.

**빌드 트러블슈팅:** build-error-resolver agent 사용 → 오류 분석 → 점진적 수정 → 각 수정 후 검증.
